AWS Cloud WAN Centralized Egress Lab

Step-by-step manual for building, validating and explaining the lab from zero.

Goal: Build an AWS networking lab that demonstrates Cloud WAN core networks, global edge locations, VPC attachments, segment policies, service insertion, AWS Network Firewall, NAT egress and CloudWatch validation.
Cost warning: This is a short-lived demo lab. AWS Network Firewall, NAT Gateways and Cloud WAN attachments can become expensive if left running. Destroy the lab after screenshots and validation.

1. Architecture Summary

The lab deploys a Cloud WAN global network with three core network edge locations:

CodeRegionAWS regionPurpose
IADNorthern Virginiaus-east-1Workload and inspection VPC
PDXOregonus-west-2Workload and inspection VPC
DUBIrelandeu-west-1Workload and inspection VPC plus validation host
Private EC2 workload
  -> workload private route table
  -> AWS Cloud WAN core network
  -> Prod segment service insertion
  -> InspectionNFG network function group
  -> AWS Network Firewall endpoint
  -> NAT Gateway
  -> Internet Gateway
  -> Internet

This lab does not use a Transit Gateway. VPCs attach directly to the Cloud WAN core network.

Global Network overview showing three Cloud WAN edge locations and zero Transit Gateways.

2. Prerequisites

  • AWS account with permissions for Cloud WAN, VPC, EC2, IAM, Network Firewall, NAT Gateway and CloudWatch Logs.
  • Terraform installed locally.
  • AWS CLI configured for the target account.
  • SSM Session Manager support. The lab creates IAM roles for the private validation EC2 instances.

3. Deploy From Zero With Terraform

Step 1 - Open the Terraform folder

cd C:\Users\davidgo2\Downloads\aws-cloudwan-centralized-egress-lab\terraform

Step 2 - Initialize Terraform

terraform init

Step 3 - Choose the lab profile

The 3-region demo variable file is:

..\examples\current-3region-lab.tfvars.example

It enables Cloud WAN, service insertion, AWS Network Firewall, NAT Gateways and private SSM validation instances.

Step 4 - Plan

terraform plan "-var-file=..\examples\current-3region-lab.tfvars.example" "-out=cloudwan-3region.tfplan"

Step 5 - Apply

terraform apply "cloudwan-3region.tfplan"

Cloud WAN policy updates can take several minutes. Network Firewall endpoints can also take time to become ready.

5. Cloud WAN Policy

Cloud WAN classifies attachments through tags and attachment policies.

Attachment policies map inspection attachments to InspectionNFG and workload attachments to Prod.

The functional tags are:

Attachment typeTagResult
Workload VPC attachmentsegment = ProdAssociated to the Prod segment.
Inspection VPC attachmentnetwork-function-group = InspectionNFGAdded to the inspection network function group.

The InspectionNFG network function group represents regional inspection VPC attachments.

The segment action sends Prod traffic to the InspectionNFG service insertion target.

The attachment list shows workload VPCs in Prod and inspection VPCs mapped to the network function group.

6. VPC Attachments

Workload VPC attachments do not use appliance mode. They only send workload traffic to the Cloud WAN core.

DUB workload VPC attachment using the private workload subnet.

Inspection VPC attachments do use appliance mode because they host firewall infrastructure.

Inspection VPC attachment with appliance mode enabled.

7. Segments

The Prod segment spans all three edge locations. Isolated attachments are enabled to avoid broad lateral connectivity between workload attachments.

The Prod segment across IAD, PDX and DUB.

8. Route Tables

The route tables prove the data path. In DUB, the workload VPC has one private route table, and the inspection VPC has route tables for Cloud WAN attachment traffic, firewall subnet traffic and public/NAT egress.

DUB route tables created by Terraform.

Workload private route table

The private workload subnet sends default traffic to the Cloud WAN core network.

10.30.0.0/16  -> local
0.0.0.0/0     -> core-network-0504385422771f617

DUB workload private route table default route to Cloud WAN.

Inspection Cloud WAN route table

Traffic entering the inspection VPC from Cloud WAN is sent to the Network Firewall endpoint.

100.64.30.0/24 -> local
0.0.0.0/0      -> Network Firewall endpoint

DUB inspection Cloud WAN route table default route to firewall endpoint.

Inspection firewall route table

After inspection, internet-bound traffic goes to NAT Gateway. Workload CIDRs return to Cloud WAN.

0.0.0.0/0    -> NAT Gateway
10.10.0.0/16 -> Cloud WAN core
10.20.0.0/16 -> Cloud WAN core
10.30.0.0/16 -> Cloud WAN core

DUB firewall subnet route table with NAT egress and Cloud WAN return paths.

Inspection public route table

The public subnet reaches the internet through the IGW and sends workload CIDR return traffic back through the firewall endpoint.

0.0.0.0/0    -> Internet Gateway
10.10.0.0/16 -> Network Firewall endpoint
10.20.0.0/16 -> Network Firewall endpoint
10.30.0.0/16 -> Network Firewall endpoint

DUB public route table keeps return traffic symmetric through the firewall endpoint.

10. Private EC2 Validation

The validation host is private and managed with SSM. It does not require SSH or a public IP.

DUB private validation instance in the workload VPC.

Run the SSM command

In Systems Manager Run Command, use document AWS-RunShellScript and target instance i-0ae36a72cfb42e7a8.

hostname
ip route
curl -s -I -m 10 https://aws.amazon.com | head -n 1
getent hosts amazon.com | head -n 3

Expected result:

ip-10-30-10-236.eu-west-1.compute.internal
default via 10.30.10.1 dev ens5 ...
HTTP/2 200
98.x.x.x amazon.com

SSM Run Command confirms that the private EC2 instance reached aws.amazon.com.

11. CloudWatch Flow Log Validation

Use CloudWatch Logs Insights in eu-west-1 against this log group:

/aws/network-firewall/cloudwan-egress-lab-cheap-dub-inspection/flow

Query:

fields @timestamp, event.src_ip, event.dest_ip, event.dest_port, event.app_proto, event.proto
| filter event.src_ip = "10.30.10.236" or event.dest_ip = "10.30.10.236"
| sort @timestamp desc
| limit 20

Expected row:

event.src_ip     = 10.30.10.236
event.dest_port  = 443
event.app_proto  = tls
event.proto      = TCP

CloudWatch Network Firewall flow logs prove TLS/443 traffic from the private workload IP.
