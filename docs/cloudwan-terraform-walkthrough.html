AWS Cloud WAN Terraform Walkthrough

Purpose: Explain the Terraform-built project end to end using the captured evidence from the live lab. The screenshots are used once each in this document.

The project builds a three-region Cloud WAN centralized egress design with workload VPCs, inspection VPCs, service insertion, AWS Network Firewall, NAT Gateways, private EC2 validation hosts and CloudWatch flow logs.

1. Terraform Project Shape

PathRole
terraform/main.tfRegional spoke and inspection modules.
terraform/cloudwan.tfGlobal Network, Core Network, policy attachment and VPC attachments.
terraform/routes.tfRoute table programming for workload, firewall, NAT and Cloud WAN return paths.
terraform/locals.tfRegion map, tags and Cloud WAN policy JSON composition.
terraform/modules/regional-spokeWorkload VPC, private subnet and optional SSM validation instance.
terraform/modules/inspection-vpcInspection VPC, AWS Network Firewall, NAT Gateway, IGW and logging.

3. Cloud WAN Global Network

Terraform creates the global network and deploys a core network edge in each enabled region.

01 - Cloud WAN global network with three edge locations and zero Transit Gateways.

4. Cloud WAN Policy Logic

The policy classifies attachments through tags. This avoids manual one-off mapping and makes the design explainable.

02 - Attachment policies map inspection attachments to `InspectionNFG` and workload attachments to `Prod`.

03 - `InspectionNFG` groups regional inspection VPC attachments.

04 - Segment action sends `Prod` egress traffic to the inspection network function group.

05 - Final attachment mapping: inspection VPCs in the NFG and workload VPCs in `Prod`.

5. VPC Attachments

Workload VPC attachments connect application subnets to Cloud WAN. Inspection attachments are special because they host firewall endpoints and require appliance mode.

06 - DUB workload VPC attachment selects the workload private subnet.

07 - Inspection VPC attachment uses appliance mode support.

08 - `Prod` segment is available across IAD, PDX and DUB.

6. DUB Route Tables

DUB is the validation region. The packet path is visible in its route tables.

09 - DUB route table inventory.

10 - Workload private subnet sends `0.0.0.0/0` to the Cloud WAN core network.

11 - Traffic entering the inspection VPC from Cloud WAN goes to the firewall endpoint.

12 - Firewall subnet sends internet traffic to NAT and return traffic to Cloud WAN.

13 - Public/NAT subnet sends internet traffic to IGW and return workload traffic to the firewall endpoint.

7. AWS Network Firewall

The inspection VPC contains the managed firewall. Terraform creates the rule group, policy, firewall endpoint and logging configuration.

14 - Stateful Suricata-compatible egress rules for HTTP, HTTPS and DNS.

15 - Firewall is ready and in sync.

16 - NAT Gateway is available for inspected egress.

8. Private Validation

The validation instance lives in a private subnet with no public IP. SSM is used instead of SSH.

17 - Private EC2 validation instance in DUB workload VPC.

18 - SSM Run Command confirms `HTTP/2 200` from `aws.amazon.com`.

19 - CloudWatch Logs Insights confirms TLS/443 traffic from the private workload IP.

20 - Governance and cost tracking tags.

10. Project Summary

I built a Terraform-based AWS Cloud WAN centralized egress lab across three regions.
The project maps enterprise WAN concepts to AWS-native objects: Cloud WAN core network edges,
segments, attachment policies, network function groups, service insertion, inspection VPCs,
AWS Network Firewall, NAT egress and CloudWatch validation. I validated the path from a private
EC2 instance using SSM and confirmed the inspected TLS flow in Network Firewall logs.
